# 后端架构

> **权威性说明：** 本文件是 `doc/backend/backend-architecture.md` 的中文镜像，只供人工阅读。AI agent 和实现工作必须以英文 `doc/` 为准。

Phase 1 后端是 Spring Boot 模块化单体。

## 架构目标

- Phase 1 保持部署简单。
- 模块边界足够严格，便于未来拆分。
- 尽量避免业务规则与框架深度耦合。
- 让存储、搜索、邮件和校园登录供应商可替换。

## 模块

推荐后端模块：

- `auth`：邮箱登录、校园扫码登录、会话、身份可信等级、授权辅助能力。
- `user`：用户资料、账号状态、会员展示状态、设置。
- `content`：统一内容模型、板块、标签、发布。
- `comment`：评论、回复、评论治理、评论互动。
- `review`：资源认证、置顶内容审核、公告审核、治理任务。
- `file-storage`：上传、下载、对象存储适配器、附件元数据。
- `notification`：站内通知和重要邮件通知。
- `search`：Elasticsearch 索引和搜索 API。
- `admin`：协调其他模块的管理工作流。
- `audit`：管理员操作、安全日志和敏感操作日志。

## 模块边界规则

- 按业务能力拆分，不只按技术层拆分。
- 模块可以向其他模块暴露应用服务或接口。
- 不要直接访问其他模块的内部 repository、entity 或实现类。
- 每个模块拥有自己的表。跨模块读取需要文档化的查询服务、投影或读模型。
- 跨模块写入必须通过所属模块的应用服务。

## 框架隔离

- 业务规则应尽可能使用普通 Java 代码表达。
- HTTP DTO、ORM Entity 和框架注解不能成为业务规则的唯一表示。
- 供应商 SDK 必须隔离在适配器后面。

## 未来 Axum 迁移

Rust Axum 是未来选项，不是 Phase 1 依赖。

适合拆分的候选：

- `file-storage`
- `search`
- `notification`

不适合一开始拆分的候选：

- 领域尚不稳定前的共享内容规则。
- 校园登录和权限规则尚不稳定前的 auth。
